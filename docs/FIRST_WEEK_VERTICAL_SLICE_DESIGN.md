# 第一週垂直切片：鎖定企劃規格（v1.1 草案）

> 狀態：待確認。這份文件確認前，不再擴充新的地點、角色或長期系統。
>
> 目的：讓「第一週」成為一個可從開局玩到結果、能看懂選擇後果的完整小遊戲；它不是功能清單，也不是長篇世界觀設定。

## 1. 這一版要交付什麼

玩家從第 1 天開始，能自由安排既有的接客、休息、商店與探索行動；第 7 天晚上一定面對房東結算，並依資金與阿坤的協助得到不同結果。每個重要數值都必須由可見的行動改變，且在事件或結算中產生可見後果。

本切片完成的判斷只有一個：新玩家不看外部說明，也能知道「現在要做什麼、為何要做、這次選擇改變了什麼」。

### 不可變原則

1. **自由不是沒有方向。** 玩家每天可自行選行動；期限與事件只提供壓力、資訊與後果。
2. **文本不是裝飾。** 每張事件文本至少有一個可觀察的系統結果，不能只換一句敘述。
3. **每個數值都要可追溯。** 玩家可以從日誌、事件結果或狀態列知道數值何時、為何改變。
4. **第一週不做失敗即刪檔。** 未達標是較困難的下一章起點，不中斷存檔也不強迫重玩。
5. **先完成閉環，再增加內容。** 沒有接上觸發、後果與驗收的地點、角色、數值都不加入此版本。

## 2. 範圍與非範圍

### 本版本包含

- 第 1 至第 7 天的期限、目標、事件與結算。
- 現有核心行動接到第一週規則：接客、休息、商店、移動、存檔。
- 房東是期限與收租的壓力來源；阿坤是商店店長，信任值代表他是否願意以工作機會或預支協助玩家。好感與支配先保留資料欄位，不在本切片判定結果。
- 5 張固定事件卡與 3 組章末結果文本。
- 目標面板、事件畫面、結算畫面、結果摘要與下一週狀態。
- 舊存檔相容與最少的自動測試。

### 明確不做

- 新的長期地圖、額外可進入地點、完整第二週故事。
- 新的身體、生理、裝備或複雜經營系統。
- 任意大量隨機事件池；第一週先以固定事件驗證節奏。
- 用大量新文本掩蓋尚未成立的選擇後果。

## 3. 玩家體驗與核心循環

### 玩家在第一週應學到的事

| 問題 | 玩家在何處得到答案 | 系統證據 |
| --- | --- | --- |
| 我要做什麼？ | 目標面板、開場事件 | 1200G 與第 7 天期限 |
| 為何不能一直接客？ | 體力下降、休息教學 | 體力限制後續行動或效率 |
| 金錢以外有何用？ | 商店與結算 | 裝備／工作機會／資金門檻 |
| 為何理會阿坤？ | 第 3、5 天事件與第 7 天 | 信任不足與信任達標的結算不同 |
| 我能不能照自己的方式玩？ | 日常自由行動 | 同一結果可由不同日程達成 |

### 可玩循環

```text
讀取目標 → 選擇日常行動 → 時間與數值改變
  → 檢查是否有待觸發事件 → 做出事件選擇 → 顯示後果
  → 跨日 → 更新目標與日誌 → 第 7 天晚上結算
```

**重要：** 事件不能在行動按下前憑空跳出。它必須由「跨日、進入特定時段、第一次完成某行動、或滿足明確條件」之一觸發；畫面必須告訴玩家觸發原因。

## 4. 時間、章節與事件的技術規則

現有 `addMinutes(player, mins)` 已負責跨日。第一週功能不可在每個按鈕各自判斷日期，必須用同一層章節檢查避免遺漏。

### 4.1 狀態模型（目標資料契約）

```js
progress: {
  chapter: 'first_week',
  phase: 'tutorial', // tutorial | active | closing | resolved
  deadlineDay: 7,
  targetGold: 1200, // 第 7 天需要交給房東的房租／欠租總額
  tutorialStep: 0,
  customerVisits: 0,
  resolvedEventIds: [],
  pendingEventId: null,
  outcomeId: null,
  resolved: false,
},

relationships: {
  shopManager: { affection: 0, trust: 0, dominance: 0 },
},

flags: {
  firstWeekShopJobSeen: false,
  firstWeekBossPromiseMade: false,
  firstWeekOutcomeSeen: false,
}
```

規則：

- `pendingEventId` 不為空時，日常行動按鈕鎖定，只能閱讀並完成事件。
- `resolvedEventIds` 是唯一的「這張固定事件已做過」來源，不以日誌文字判斷。
- `outcomeId` 寫入後不可被之後的自動邏輯覆蓋。
- 載入舊存檔時，缺少欄位以預設值補齊；不得重新播放已完成教學。

### 4.2 單一入口

新增一個章節服務層（名稱可為 `progression` 或 `chapterEngine`），提供下列純函式：

| 函式 | 輸入 | 回傳／責任 |
| --- | --- | --- |
| `advanceChapterTime` | player, minutes | 呼叫既有時間計算後，找出跨日或時段造成的待處理事件 |
| `getPendingFirstWeekEvent` | player | 依固定優先順序回傳一張事件或 `null` |
| `applyFirstWeekChoice` | player, choiceId | 回傳新的 player、關係、日誌訊息與事件結果 |
| `resolveFirstWeek` | player, relationships | 回傳唯一的章末結果 |
| `getFirstWeekObjective` | player | 回傳畫面可直接顯示的目前目標 |

所有第一週的日常行動應改由 `advanceChapterTime` 進入時間系統；第一階段只替換接客、休息、商店、移動這四類高頻行動。其餘既有小系統先保留，但必須在驗收時確認跨日不會跳過結算。

### 4.3 事件檢查優先順序

一次只允許一張事件待處理，優先順序固定如下：

1. 第 7 天結算（截止時優先於所有事件）。
2. 第 5 天阿坤提案。
3. 第 3 天商店工作／信任選擇。
4. 第一次進入商店的工作介紹。
5. 開場通知。

同一天跨過多個時段時，只排入最高優先事件；玩家完成後再檢查下一張，避免連續彈窗淹沒日常玩法。

### 4.4 截止的精確規則

- 第 7 天 21:00 起，第一次完成任何會推進時間的行動後，排入 `week_settlement`。
- 玩家若在第 7 天 21:00 前主動休息至隔日，跨日後也必須先結算，不能直接跳到第 8 天日常。
- 結算前不凍結玩家的其他行動；21:00 前仍可自行決定是否衝最後一次收入或保留體力。
- 結算完成後設定 `phase: 'resolved'`，目標面板改顯示下一章入口，不再顯示第一週倒數。

## 5. 數值設計：來源、消耗與可見後果

第一週僅承諾下表四個可感知數值。其他既有數值可以存在，但不以新文本宣稱它們在第一週有重大影響。

| 數值 | 主要來源 | 主要消耗／風險 | 第一週可見後果 | 顯示方式 |
| --- | --- | --- | --- | --- |
| 金錢 | 接客、商店工作、事件報酬 | 購物、部分事件成本 | 1200G 達標與獨立起點 | 狀態列、目標面板、事件結果 |
| 體力 | 休息 | 接客、移動、服務行動 | 決定玩家是否能安全持續行動 | 狀態列、行動不足提示 |
| 名聲 | 既有接客成果與指定事件 | 不在第一週直接扣除 | 僅改變客人品質與日誌敘事；**不**作為結算門檻 | 狀態列、客人生成說明 |
| 阿坤信任 | 第 3、5 天選擇、商店工作 | 拒絕承諾或失約（只在明確事件中） | 未達資金目標時，阿坤能否提供工作證明／預支，協助向房東爭取延期 | 事件結果、結算摘要 |

### 金錢平衡規則

- 1200G 是暫定目標，不在本次直接調整既有接客報酬。
- 完成實作後，要用三種玩法各跑 5 次：保守休息、平均接客、衝收入。
- 平均玩法達標率目標為 40% 至 70%；低於 40% 降目標或增加可預期收入，高於 70% 增加成本或降低穩定收入。
- 平衡改動必須記錄在本文件的「平衡紀錄」；不以感覺偷偷改數字。

## 6. 固定事件卡

事件文本採「短敘事＋選項＋立即後果＋延後伏筆」。第一週每張事件最多三個選項，避免第一次體驗出現假自由或過多分支。

### E0：開場通知

| 欄位 | 規格 |
| --- | --- |
| 觸發 | 新遊戲建立後、任何日常行動前；只觸發一次 |
| 目的 | 說明第 7 天期限、1200G 目標與「能自由安排日程」 |
| 文本功能 | 不介紹所有系統，只給玩家第一個可做的行動：接第一位客人或去商店看看 |
| 選項 | 「先接一位客人」／「先去商店」；兩者不改數值，只導向既有畫面 |
| 後果 | 寫入 `resolvedEventIds: ['opening_notice']`，`phase` 進入 `tutorial` |

### E1：第一次商店工作介紹

| 欄位 | 規格 |
| --- | --- |
| 觸發 | 第一週第一次進入商店；若到第 3 天仍未進商店，改在第 3 天第一個跨日後觸發 |
| 目的 | 告訴玩家商店是較低風險、低報酬、可建立關係的替代選擇 |
| 選項 | 詢問工作／只購物離開 |
| 即時效果 | 詢問工作：`firstWeekShopJobSeen=true`，不強制接工作；只購物：無懲罰 |
| 文本伏筆 | 阿坤提到第 3 天可再談一件跑腿事 |

### E2：第 3 天的信任選擇

| 欄位 | 規格 |
| --- | --- |
| 觸發 | `days >= 3`、未完成、時間為早上或下午；若玩家未去商店，事件仍可從日誌／訊息入口出現 |
| 前提 | 不要求商店工作介紹已完成，避免玩家因探索順序錯過結算路線 |
| 選項 A：幫忙 | 消耗 90 分鐘，`shopManager.trust +1`，獲得少量固定金錢或商店折扣；明確顯示兩者數值 |
| 選項 B：談條件 | 消耗 45 分鐘，若名聲或金錢達一個低門檻則 `trust +1` 並得到較少報酬；未達門檻則無信任且不懲罰 |
| 選項 C：婉拒 | 不消耗時間，無信任改變；文本明確說明之後仍可在第 5 天取得一次信任機會 |
| 後果 | 寫入 `firstWeekBossPromiseMade` 僅代表玩家已知悉關係路線，不可當作信任值替代品 |

### E3：第 5 天的期限提案

| 欄位 | 規格 |
| --- | --- |
| 觸發 | `days >= 5`、未完成、尚未進入結算 |
| 目的 | 讓玩家在看見目前資金後，主動選擇繼續衝金錢或投資延期保險 |
| 選項 A：承諾完成工作 | 消耗 120 分鐘，`shopManager.trust +1`；若體力過低，按鈕顯示不可用原因，不可偷偷扣體力 |
| 選項 B：專心賺錢 | 不改信任，給出目前距離目標的精確提示 |
| 選項 C：要求緩衝 | 僅在 `shopManager.trust >= 1` 時可選；改成一個小成本／少量金錢交換，並標示它不保證達標 |
| 後果 | 玩家即使拒絕也不被鎖死；信任路線只有在未達 1200G 時才會真正發揮價值 |

### E4：第 7 天結算

| 欄位 | 規格 |
| --- | --- |
| 觸發 | 依第 4.4 節，強制但不可早於第 7 天 21:00 |
| 畫面 | 先列出「資金、目標、信任、做過的關鍵選擇」，再顯示結果標題與文本 |
| 選項 | 僅「繼續」；結果由之前行動決定，不能在此時補救數值 |
| 寫入 | `outcomeId`、`phase='resolved'`、`resolved=true`、`firstWeekOutcomeSeen=true` |

## 7. 三種結算與後續狀態

結算採**固定判定順序**，避免文字與系統衝突。

| 優先 | 結果 ID | 條件 | 玩家得到的意義 | 下一章初始狀態 |
| --- | --- | --- | --- | --- |
| 1 | `independent_start` | `gold >= targetGold` | 靠自己的日程與收入達標 | 解鎖「自主起點」標記；保留金錢與既有名聲 |
| 2 | `conditional_extension` | 未達資金且 `shopManager.trust >= 2` | 阿坤以工作證明或預支協助玩家向房東爭取有限期限 | 解鎖「延期」標記；下一週顯示新的較短目標與阿坤要求 |
| 3 | `debt_deepens` | 其他所有情況 | 沒有達標也沒建立可求助的關係 | 解鎖「債務加深」標記；下一週保留自由行動但提高壓力敘事 |

### 文本規格

- 每種結果 120 至 180 個中文字，分成 2 至 3 段，不使用純旁白結論代替玩家行動。
- 每種結果必須點名兩個可追溯證據，例如「你帶回了 XG」或「你完成了某次工作」。
- 文字不批判玩家選擇；未達標文本要傳達更高壓力，而非「你玩錯了」。
- 結算畫面要有「本週摘要」：接客次數、休息次數、商店互動次數、信任變化與金錢差額。缺少統計時，寧可不顯示，不可編造。

## 8. UI 與資訊層級

### 常駐目標面板

面板只顯示一個「現在最重要」的訊息，優先序為：待處理事件 > 教學步驟 > 截止金錢目標 > 已結算的下一步。

| 狀態 | 標題 | 必顯示資料 | 禁止顯示 |
| --- | --- | --- | --- |
| 教學 | 下一個具體動作 | 一句原因、可去的地點 | 長篇世界觀 |
| 日常 | 第一週：建立立足點 | 第幾天、差多少 G、下一個事件提示 | 未來章節資訊 |
| 結算待處理 | 今晚必須結算 | 資金、信任、結算按鈕 | 新的日常行動 |
| 已結算 | 第一週結果 | 結果名稱、下一章入口 | 失效的倒數 |

### 事件畫面

1. 標題：誰／什麼事正在發生。
2. 正文：60 至 120 字，先說情境再說壓力。
3. 選項：每個按鈕標示時間、金錢、體力或信任的可預期成本；未知機率才可標示「可能」。
4. 結果：先顯示數值變化，再給 1 至 3 句反應文本。
5. 返回：事件結果確認後才回主畫面並刷新目標面板。

### 美術資產與畫面用途

美術是可玩資訊的一部分：玩家必須從色彩、構圖與角色表情分辨「安全的日常」、「可交換的人情」與「無法逃避的收租壓力」。本版本不以大量角色或每段劇情都配 CG 為目標；每一張圖都要對應一個不可替代的玩法節點。

#### 視覺語言

| 場景／角色 | 主色 | 玩家應立刻讀到的意義 | 構圖規則 |
| --- | --- | --- | --- |
| 玩家房間 | 灰藍、低飽和暗紅 | 私人空間、疲憊、租期壓力 | 留出右側給角色立繪與訊息卡 |
| 阿坤商店 | 暖黃、墨綠 | 穩定但需要人情交換的選項 | 櫃台形成水平線，阿坤在左、選項在右 |
| 街道／工作地點 | 紫紅、霓虹反光 | 高報酬與較高風險 | 避免與商店同色，讓玩家一眼辨識 |
| 房東相關畫面 | 冷白、深棕、硬陰影 | 期限、帳單、被迫面對的結果 | 房東不必完整露臉；帳單、門框與視線壓迫要明確 |

#### 主角立繪規格

| 資產 | 數量 | 必須傳達的狀態 | 使用位置 |
| --- | ---: | --- | --- |
| 主角基礎全身立繪 | 1 | 能對應日常服與一般對話 | 主畫面、日常事件 |
| 表情差分 | 4 | 平靜／戒備／疲憊／下定決心 | 依事件結果與體力狀態切換 |
| 服裝差分 | 2 | 日常服、工作服 | E0、商店事件與日常選擇 |
| 近景頭像 | 1 | 供結算與重要內心反應使用 | 第 7 天結算、結果摘要 |

- 成品尺寸：透明背景 PNG，直式 1536 × 2048；頭像可由同一來源裁切，不另畫第二張。
- 程式先只支援「服裝 ID + 表情 ID」，不在第一週嘗試讓所有衣櫃物件逐件改變立繪。
- 主角的疲憊表情只能由低體力或事件結果觸發，不可隨機切換；這是數值可視化的一部分。

#### 特殊事件 CG 清單

| ID | 觸發事件 | CG 內容與鏡頭 | 必須支援的玩法效果 | 可否暫用占位圖 |
| --- | --- | --- | --- | --- |
| `cg_opening_rent_notice` | E0 開場通知 | 房間、催繳通知、主角近景 | 讓玩家理解房東與第 7 天壓力 | 可以；需保留文字安全區 |
| `cg_akun_day3` | E2 第 3 天工作選擇 | 商店櫃台、阿坤遞出工作／帳本 | 讓「花時間換信任」有記憶點 | 可以 |
| `cg_akun_day5` | E3 第 5 天期限提案 | 阿坤與主角、剩餘資金提示 | 清楚表達此刻是在選擇金錢路線或人情路線 | 可以 |
| `cg_outcome_independent` | 達成 1200G | 主角在房間整理帳單、光線轉暖 | 成功繳租且保留主動性 | 不可；第一版正式展示必備 |
| `cg_outcome_extension` | 信任協助延期 | 主角與阿坤、工作證明／預支單據 | 協助有代價但不是被拯救的結局 | 可以，但公開測試前需完成 |
| `cg_outcome_debt` | 債務加深 | 房門、帳單與主角的決心近景 | 壓力提高但故事繼續 | 可以，但不得用純黑畫面取代 |

CG 成品尺寸為 1920 × 1080 PNG，文字放在畫面下方 30% 的安全區以外；同一事件的文字不得依賴 CG 裡的小字才能被理解。所有人物、場景與服裝在同一張 CG 中要與主角立繪色彩及外觀一致。

#### 美術製作閘門

1. **灰盒階段**：先用標準場景卡與灰階人物剪影，驗證事件流程、文字安全區與按鈕位置。
2. **可玩 Alpha**：完成主角基礎立繪、4 表情、房間／商店背景與開場 CG；此時即可讓外部測試者完整玩一週。
3. **內容 Beta**：完成兩張阿坤事件 CG 與三張結果 CG；檢查每種結算都有相對應畫面。
4. **發布前**：逐張在 16:9 桌面與窄螢幕測試，確保 CG 不遮住選項、文字對比符合閱讀需求。

## 9. 實作順序與提交邊界

每一階段都必須可獨立測試；沒有通過前不得開始下一階段。

1. **資料與遷移**：補齊 `phase`、事件 ID、結果 ID、旗標；新舊存檔均可載入。
2. **章節引擎**：建立單一時間入口與待處理事件檢查；先不寫長文本。
3. **事件 UI**：能顯示一張假事件、阻擋日常行動、套用選項結果、回到主畫面。
4. **5 張事件卡與結算**：依第 6、7 節加入真實文本與數值效果。
5. **摘要與下一週狀態**：完成結果持久化與 UI。
6. **平衡與可玩性測試**：依第 5 節的三種玩法記錄結果，才調整 1200G 或報酬。

每次提交只處理一個階段；不可在同一提交混入新地點、美術或不相關重構。

- [ ] 三種結算條件各有自動測試，且判定順序正確。
- [ ] 舊存檔可以載入並取得安全預設；沒有既有進度的存檔不會報錯。
